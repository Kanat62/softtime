# SoftTime — SaaS-платформа управления офисом (Основной ТЗ)

# ТЕХНИЧЕСКОЕ ЗАДАНИЕ v3.0 (Мультитенант / SaaS — Июнь 2026)

> **Компания (tenant)**, **уникальный код компании** для присоединения сотрудников и **подписка/оплата** (платит только ADMIN). Все данные изолируются по `company_id`.
>

## РОЛИ ПОЛЬЗОВАТЕЛЕЙ

```
PROVIDER → ADMIN → WORKER
```

| Роль | Уровень | Описание |
| --- | --- | --- |
| PROVIDER | Платформа (мы) | Владелец проекта. Видит **все компании**, их сотрудников и оплату/подписки. Может активировать/приостанавливать компании. Не привязан к одной компании. |
| ADMIN | Компания (tenant) | Главный конкретной компании (**один на компанию**). **Единственный, кто платит подписку** (помесячно). Принимает/отклоняет сотрудников, управляет компанией, расписаниями, новостями, отчётами — **только в рамках своей компании**. |
| WORKER | Компания (tenant) | Обычный сотрудник. Регистрируется, вводит **уникальный код компании**, ждёт подтверждения ADMIN. После одобрения может вести приход/уход. |

**Изоляция данных (multi-tenancy):**

- ADMIN и WORKER принадлежат **ровно одной** компании (`company_id`).
- У каждой компании **ровно один ADMIN**; роли пользователей не меняются (нельзя назначить второго ADMIN).
- PROVIDER видит данные **всех** компаний (cross-tenant).
- Все запросы ADMIN/WORKER автоматически ограничиваются их `company_id` на уровне backend.

| Статус | Описание |
| --- | --- |
| PENDING | Зарегистрировался, ввёл код компании, ждёт подтверждения ADMIN этой компании |

| DELETED | Мягкое удаление (данные сохраняются в БД для истории) |

## СТАТУСЫ КОМПАНИИ / ПОДПИСКИ

**Статус компании (company.status):**

| Статус | Описание |
| --- | --- |
| TRIAL | Пробный период — первый месяц бесплатно, компания работает полностью |
| ACTIVE | Подписка оплачена, компания работает |
| GRACE | Льготный период после окончания пробного/оплаченного периода — работает, идут напоминания об оплате |
| SUSPENDED | Подписка не оплачена — функции заблокированы (сотрудники не могут отмечать приход/уход) |

**Статус подписки (subscription.status):** `TRIAL` | `ACTIVE` | `EXPIRED` | `GRACE` | `CANCELLED`

| Статус | Описание | Что происходит |
| --- | --- | --- |
| TRIAL | Первый месяц бесплатно для новой компании | Компания работает полностью; до окончания пробного периода оплата не требуется |
| ACTIVE | Подписка оплачена и действует | Всё работает в штатном режиме |
| GRACE | Срок оплаты прошёл (после пробного или оплаченного периода), идёт льготный период (например, 7 дней) | Компания работает в обычном режиме, ADMIN получает ежедневные push-уведомления с напоминанием об оплате |
| EXPIRED | Льготный период истёк, оплата так и не поступила | Компания переходит в статус SUSPENDED — сотрудники не могут отмечать приход/уход, ADMIN видит экран «подписка не оплачена» |
| CANCELLED | ADMIN вручную отменил подписку или PROVIDER отменил со своей стороны | Компания переходит в SUSPENDED, восстановление только через новую оплату |

## СТАТУСЫ ПОСЕЩАЕМОСТИ

**Приход (check_in_status):** `ON_TIME` | `LATE` | `EARLY_ARRIVAL`

**Уход (check_out_status):** `ON_TIME` | `LEFT_EARLY` | `OVERTIME`

**Общий статус дня (status):** `PRESENT` | `LATE` | `ABSENT` | `INCOMPLETE` | `APPROVED_ABSENCE` | `MANUAL` | `EARLY_LEAVE` | `OVERTIME`

| Статус | Описание |
| --- | --- |
| ON_TIME | Пришёл вовремя (±5 мин от начала графика) |
| LATE | Пришёл позже начала графика — фиксируется опоздание на X мин/ч |
| EARLY_ARRIVAL | Пришёл раньше начала графика — фиксируется на X мин/ч раньше |

| Статус | Описание |
| --- | --- |
| ON_TIME | Ушёл вовремя (±5 мин от конца графика) |
| LEFT_EARLY | Ушёл раньше конца графика — фиксируется ранний уход на X мин/ч |
| OVERTIME | Ушёл позже конца графика — фиксируется переработка на X мин/ч |

| Статус | Описание |
| --- | --- |
| PRESENT | Сотрудник пришёл и ушёл в рамках графика, всё в порядке |
| LATE | Сотрудник пришёл с опозданием |
| ABSENT | Сотрудник не появился в рабочий день без какой-либо причины |
| INCOMPLETE | Сотрудник пришёл, но не сделал check-out до времени автозакрытия — система проставила время ухода = конец смены, день помечен `INCOMPLETE` |
| APPROVED_ABSENCE | Отсутствие одобрено админом (отпуск, больничный и т.д.) |
| MANUAL | Запись создана или исправлена вручную администратором |
| EARLY_LEAVE | Сотрудник ушёл раньше конца рабочего дня |
| OVERTIME | Сотрудник задержался после конца рабочего дня |

## МОДУЛЬ 1 — ПЛАТФОРМА, КОМПАНИИ И ПОДПИСКА

### 1.1 Мультитенантность

- Каждая компания — изолированный tenant. У всех таблиц с данными компании есть `company_id`.
- Backend на каждом запросе ADMIN/WORKER подставляет `company_id` из токена и фильтрует данные — компания **не может** видеть чужие данные.
- PROVIDER — единственная роль с доступом ко всем компаниям.

### 1.2 Регистрация компании (онбординг ADMIN)

- ADMIN регистрирует компанию **через мобильное приложение или через веб-панель** — оба варианта равнозначны. Поля: название компании, ФИО владельца, email, пароль.
- Система создаёт запись `companies` + пользователя с ролью ADMIN, привязанного к ней.
- Генерируется **уникальный код компании** (`company_code`).
- Новая компания получает **первый месяц бесплатно** (пробный режим, статус `TRIAL`). После окончания пробного месяца — льготный период (`GRACE`) с напоминаниями об оплате; если оплата так и не поступила → `SUSPENDED`. Компания может оформить платную подписку **в любой момент** (→ `ACTIVE`).

> Компания заводится самостоятельно (self-serve).
> 

### 1.3 Уникальный код компании

- Каждая компания имеет уникальный `company_code` — **ровно 6 символов**.
- WORKER вводит этот код при регистрации, чтобы привязаться к компании.
- Код генерируется один раз при создании компании; ADMIN видит его в своей панели, чтобы передавать сотрудникам.

### 1.4 Подписка и оплата

- **Платит только ADMIN** — помесячная подписка за свою компанию.
- **Тариф: фиксированная цена — $30 в месяц за компанию** (единый тариф, без расчёта по числу сотрудников).
- У подписки: период (начало/конец), статус, дата следующего списания.
- Каждая оплата фиксируется в `payments` (сумма, дата, период, статус, способ).
- Жизненный цикл оплаты: `TRIAL` (первый месяц бесплатно) → `GRACE` (льгота) → `SUSPENDED` (если не оплачено). Оплата **в любой момент** → `ACTIVE`.
- В состоянии `SUSPENDED` сотрудники компании **не могут отмечать приход/уход** (вход в приложение возможен, но функции заблокированы с подсказкой «подписка не оплачена»).
- **Платёжный шлюз:** конкретный провайдер пока не выбран (тестовый режим). Платёжный слой проектируется **заранее под лёгкую интеграцию** — отдельный модуль оплаты + webhook от провайдера, чтобы подключить любой шлюз в любой момент без переделки логики.

### 1.5 Панель PROVIDER

PROVIDER через свою панель видит и управляет платформой:

- **Список всех компаний:** название, код, статус, число сотрудников, статус подписки, дата следующей оплаты.
- **Просмотр сотрудников любой компании** (read-only по умолчанию).
- **Оплаты / выручка:** все платежи по всем компаниям, фильтры по периоду и компании, суммарные показатели.
- **Активация / приостановка компании** вручную (например при споре по оплате).
- PROVIDER **не вмешивается** в операционные данные компании (посещаемость и т.д.) — только надзор и биллинг.

## МОДУЛЬ 2 — АУТЕНТИФИКАЦИЯ И ПОЛЬЗОВАТЕЛИ

### 2.1 Регистрация сотрудника (WORKER)

> Регистрация WORKER доступна **только через мобильное приложение**. На вебе сотрудники аккаунты не создают.
> 
1. Сотрудник регистрируется через мобильное приложение: ФИО, email (Gmail), пароль.
2. **Вводит уникальный код компании.**
3. Система находит компанию по коду:
    - код неверный → ошибка «компания не найдена»;
    - код верный → создаётся пользователь со статусом `PENDING`, привязанный к `company_id` этой компании.
4. ADMIN компании получает push: «Новый сотрудник ожидает подтверждения: [ФИО]».
5. ADMIN подтверждает → статус → `ACTIVE` → приложение полностью открывается (доступны приход/уход).
6. ADMIN отклоняет → сотрудник получает уведомление об отклонении.

### 2.2 Безопасность

- Пароль: bcrypt, cost factor **12**; минимальная длина — **8 символов**
- Access Token: **15 минут** (JWT) — содержит `user_id`, `role`, `company_id`
- Refresh Token: **30 дней** (хранится в Redis)
- UUID как primary key везде
- **Tenant-изоляция:** backend ограничивает выборки по `company_id` из токена (кроме PROVIDER)

### 2.3 Управление пользователями (ADMIN)

В рамках **своей компании**:

- Просмотр списка сотрудников с фильтром по статусу
- Изменение статуса сотрудника (роли не меняются — в компании один ADMIN)
- Просмотр профиля сотрудника (история посещаемости)
- Мягкое удаление (soft delete через `deleted_at`) — данные сохраняются для истории
- Добавление Admin-комментария к профилю

### 2.4 Заявки сотрудника (отсутствие и ранний уход)

Все заявки сотрудник подаёт сам через приложение, они уходят ADMIN его компании. Два типа:

**Тип 1 — Отсутствие (заранее):**

- Сотрудник указывает конкретную дату или период + причину (отпуск, больничный, личное и т.д.)
- ADMIN одобряет → выбранные дни автоматически помечаются `APPROVED_ABSENCE` в посещаемости
- ADMIN отклоняет → дни остаются обычными (если сотрудник не пришёл — `ABSENT`)

**Тип 2 — Ранний уход (на сегодня):**

- Сотрудник указывает причину и во сколько хочет уйти
- ADMIN одобряет → ранний check-out в этот день НЕ считается нарушением (не `LEFT_EARLY`), день помечается как одобренный ранний уход (`EARLY_LEAVE`, approved)
- ADMIN отклоняет → если сотрудник всё равно уйдёт раньше, зафиксируется `LEFT_EARLY`

Все заявки и решения по ним логируются в `audit_logs`.

## МОДУЛЬ 3 — ПОСЕЩАЕМОСТЬ

> Вся посещаемость привязана к `company_id`. Если компания `SUSPENDED` — приход/уход недоступны.
> 

### 3.1 Принцип верификации (двойная проверка)

Приход/уход засчитывается ТОЛЬКО при одновременном выполнении двух условий:

1. **QR-код** — совпадает с активным QR офиса компании
2. **IP-адрес** — устройство в офисной сети компании (из таблицы `office_networks`)

### 3.2 Расписание сотрудника (EmployeeSchedule)

- Расписание задаётся по дням недели (Пн–Вс)
- ADMIN может настроить график двумя способами:
    - **Индивидуально** — отдельный график для конкретного сотрудника
    - **Один для всех** — создать один график и применить его сразу ко всем сотрудникам. После применения у каждого сотрудника остаётся своя редактируемая копия — позже можно поправить индивидуально
- Каждый рабочий день содержит:
    - `is_working_day` — рабочий день или выходной
    - `start_time` + `end_time` — начало и конец смены
    - `auto_checkout_buffer` — буфер автозакрытия (доп. поле рядом со временем, **по умолчанию +1 час**)
- Минимальный рабочий день: **6 часов** (система не даёт сохранить расписание < 6 ч)
- Выходной день: `is_working_day = false`, времена и буфер не указываются

### 3.3 Приход (Check-In)

Сотрудник сканирует QR через мобильное приложение:

- Система проверяет QR + IP
- Система берёт расписание сотрудника на сегодня
- Определяет статус прихода:
    - Пришёл раньше → `EARLY_ARRIVAL` + "пришёл на X мин/ч раньше"
    - Пришёл вовремя (±5 мин) → `ON_TIME`
    - Пришёл позже → `LATE` + "опоздал на X мин/ч"
- Запись создаётся в таблице `attendance`

### 3.4 Уход (Check-Out)

Сотрудник сканирует QR при уходе:

- Система определяет статус ухода:
    - Ушёл раньше окончания графика → `LEFT_EARLY` + "ушёл на X мин/ч раньше"
    - Ушёл вовремя → `ON_TIME`
    - Ушёл позже графика → `OVERTIME` + "задержался на X мин/ч"
- Рассчитывается общее время работы
- Если на этот день одобрена заявка на ранний уход — ранний check-out засчитывается без нарушения (не `LEFT_EARLY`)

### 3.5 Автозакрытие смены

- Для каждого сотрудника на каждый рабочий день рассчитывается **время автозакрытия = `end_time` + `auto_checkout_buffer`** (по умолчанию +1 час, настраивается в расписании)
- Если к этому времени сотрудник не сделал check-out → система автоматически **ставит время ухода = `end_time` (конец смены по графику)** и **статус дня → `INCOMPLETE`**
- Системный cron периодически (например, каждые 15–30 мин) проверяет незакрытые сессии и закрывает те, у кого время автозакрытия прошло
- Гибко под любой бизнес: офис (смена до 18:00 + 1 ч = 19:00), ресторан (до 02:00 + 2 ч = 04:00), маркет с плавающими сменами — каждая компания настраивает буфер под себя
- **Исключение (Admin fix):** сотрудник сообщает о забытом сканировании → ADMIN в разделе «Сейчас в офисе» указывает реальное время ухода вручную (запись становится `MANUAL`)

### 3.6 Ручная правка (ADMIN)

- Администратор может:
    - Добавить запись об отсутствии с причиной (поле `note`)
    - Исправить время прихода/ухода (`is_manual = true`)
    - Указать время ухода для забытого check-out
- Все ручные правки логируются в `audit_logs`

### 3.7 Офисные сети

- Таблица `office_networks` — список разрешённых IP/подсетей **в рамках компании**
- ADMIN может добавлять/удалять/обновлять сети (для нескольких офисов компании)
- Полное удаление сети из системы (физическое, т.к. это конфигурация)

### 3.8 QR-код офиса

- Один активный QR на офис (или на сеть) компании
- Только ADMIN может регенерировать QR
- QR хранит зашифрованный токен, привязанный к офисной сети
- При смене офиса/сети — регенерировать QR

### 3.9 Отчёты посещаемости

Фильтры: по неделе | по месяцу | по произвольному периоду | по сотруднику

Показатели:

- Кто работал больше всего (суммарные часы)
- Кто приходил раньше / уходил позже
- Кто больше пропустил **без причины** (ABSENT / INCOMPLETE)
- Кто больше пропустил **с причиной** (APPROVED_ABSENCE)
- Кто чаще опаздывал (LATE + количество раз)
- Итоговая таблица по каждому сотруднику

**Доступ к отчётам:**

- **WORKER** — видит только свои данные.
- **ADMIN** — все отчёты **своей** компании.
- **PROVIDER** — отчёты по всем компаниям.

## МОДУЛЬ 4 — НОВОСТИ

### 4.1 Функционал

- ADMIN создаёт новость для своей компании: заголовок + текст (+ опционально фото)
- Все сотрудники компании видят ленту новостей
- Сотрудники могут только читать (без комментариев, без реакций)
- ADMIN видит: "Прочитали N из M сотрудников" (read tracking)
- Категории: одна общая лента на компанию

### 4.2 Read Tracking

- При открытии новости сотрудником → запись в `news_reads` (user_id + news_id + timestamp)
- ADMIN видит список: кто прочитал, кто нет

### 4.3 Уведомления

- При создании новости → push всем активным сотрудникам компании

## МОДУЛЬ 5 — УВЕДОМЛЕНИЯ

Push-уведомления (FCM) отправляются в следующих случаях:

| Событие | Получатель |
| --- | --- |
| Новый сотрудник ждёт подтверждения | ADMIN компании |
| Новость опубликована | Сотрудники компании |

## БИЗНЕС-ПРАВИЛА (ОБЯЗАТЕЛЬНО соблюдать)

1. **UUID** как primary key везде
2. **Multi-tenancy:** у всех данных компании есть `company_id`; ADMIN/WORKER видят только свою компанию, PROVIDER — все
3. **Подписку платит только ADMIN** — фиксированно **$30/мес за компанию**; новая компания первый месяц бесплатно (`TRIAL`), далее `GRACE → SUSPENDED` при неоплате; оплата в любой момент → `ACTIVE`
4. **В статусе SUSPENDED** приход/уход недоступны для сотрудников компании
5. **Soft delete** через `deleted_at` — физического удаления нет (кроме office_networks)
6. **Пароли** только через bcrypt (cost factor 12), минимум 8 символов
7. **JWT**: Access 15 мин, Refresh 30 дней; токен содержит `company_id`
8. **Минимум рабочего дня**: 6 часов — API возвращает ошибку при попытке сохранить расписание < 6 ч
9. **Посещаемость** = QR токен + IP офисной сети (оба фактора обязательны)
10. **Автозакрытие смены**: незакрытые сессии закрываются по времени `end_time` + буфер (по умолчанию +1 ч); система ставит время ухода = `end_time` и статус `INCOMPLETE`. Единого фиксированного времени для всех компаний нет.
11. **Все действия ADMIN/PROVIDER** логировать в таблицу `audit_logs`
12. **Мягкое удаление** пользователя — данные сохраняются для истории отчётов
13. **Код компании** — ровно 6 символов, уникален, генерируется один раз (регенерация не предусмотрена)
14. **Платёжный слой** проектируется абстрактно (модуль оплаты + webhook), чтобы подключить любой шлюз без переделки логики
