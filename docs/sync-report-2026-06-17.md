## 2. Почему у нас ломался запуск (корневая причина)

Проект — **Expo (managed)**. Это значит:

- Папки `ios/` и `android/` **не являются исходниками**. Они **генерируются** командой `npx expo prebuild` из `app.json`. Источник правды — **только `app.json`**.
- У меня генерируется `ios/` (Mac), у тебя — `android/` (Windows). Эти папки **у каждого свои** и не должны быть в репозитории.
- Если правишь файлы **внутри** `ios/` или `android/` руками — изменения **затрутся** при следующем `prebuild`. Всё нативное (иконки, права, плагины) меняется в `app.json`, а не в этих папках.

Скорее всего, ломалось из-за одной из этих причин:
- кто-то закоммитил `android/` или `ios/` → у другого конфликт/битая сборка;
- разные `.env` или захардкоженный URL попадал в git;
- после pull не делали `npm install`, и расходились версии зависимостей.

## 4. Правила, чтобы больше не ломалось

1. **Не коммить `ios/`, `android/`, `.env`.** Никогда. Они локальные.
2. **Нативные настройки** (права, плагины, иконки, bundleId) — только через `app.json`. После изменений: `npx expo prebuild` (или `npx expo prebuild --clean`).
3. **После `git pull`:** `npm install` в **корне** монорепо (workspaces ставят зависимости для всех пакетов сразу).
4. **`.env` у каждого свой.** Дефолтный URL в коде — `http://localhost:3000`. На физическом устройстве так не работает:
   - **Android-эмулятор:** `http://10.0.2.2:3000`
   - **Android физический телефон (твой случай):** IP твоего ПК в локалке, напр. `http://192.168.X.X:3000`
   - iOS-симулятор: `http://localhost:3000`
   - iPhone по кабелю (мой случай): IP моего Mac
   Шаблон со всеми вариантами уже лежит в `mobile/.env.example`.
5. **Если сборка «вдруг сломалась»** после pull — это почти всегда рассинхрон нативки. Лечится так:
   ```bash
   # из папки mobile/
   npx expo prebuild --clean   # пересоздать нативную папку из app.json
   # затем заново собрать на устройство
   npx expo run:android        # у тебя
   # npx expo run:ios          # у меня
   ```

## 5. Что нужно сделать ТЕБЕ один раз (Windows/Android)

1. `git pull`
2. В **корне** репо: `npm install`
3. Создай `mobile/.env` (его нет в git — это нормально):
   ```
   EXPO_PUBLIC_API_URL=http://10.0.2.2:3000        # если запускаешь на эмуляторе
   # или http://192.168.X.X:3000 — если на физическом телефоне (IP твоего ПК)
   ```
4. Сгенерируй нативный Android-проект:
   ```bash
   cd mobile
   npx expo prebuild -p android
   npx expo run:android
   ```
5. **Проверь, что у тебя локально НЕ закоммичены** `mobile/android/` и `mobile/.env`:
   ```bash
   git status
   ```
   Если видишь там `mobile/android/...` — не добавляй их в коммит (после фикса .gitignore из п.6 они пропадут из статуса).

## 6. ПОЧЕМУ у нас были конфликты, хотя ios/android не видны друг другу

Это главное. Native-папки тут ни при чём — они никогда не коммитились. Конфликты приходили от **общих закоммиченных файлов**, которые мы оба меняем:

### Причина №1 — `package-lock.json`
- Он **один на весь монорепо** и **закоммичен**. Менялся в **8 из последних 10 коммитов**.
- Внутри — пакеты под конкретную ОС: у меня (Mac) пишутся `@esbuild/darwin-arm64`, `fsevents`; у тебя (Windows) — `@esbuild/win32-*`.
- Каждый `npm install` на своей ОС перегенерирует lock по-своему. Оба коммитим → **конфликт в `package-lock.json`** на каждом merge.

### Причина №2 — переносы строк (CRLF vs LF)
- Windows пишет `\r\n` (CRLF), Mac — `\n` (LF).
- Без `.gitattributes` git считает весь файл изменённым → шумные диффы и конфликты даже на `.ts`/`.tsx`, где никто ничего по сути не менял.

## 7. Что сделано для лечения конфликтов

1. Добавлен **`.gitattributes`** в корень — нормализует все переносы строк к **LF**. Это убирает причину №2.
2. `.gitignore` приведён в порядок и снабжён комментариями (android/ios + артефакты сборки).

### ⚠️ Тебе (Windows) нужно сделать один раз после pull:
```bash
git pull
# перенормализовать переносы строк под новый .gitattributes:
git add --renormalize .
git status        # посмотреть, что нормализовалось
# если есть изменения — закоммить их одним коммитом:
git commit -m "chore: normalize line endings to LF"
```
Желательно сделать это, когда у обоих нет незакоммиченных правок, чтобы renormalize не смешался с реальной работой.

### Правила против конфликтов в `package-lock.json`:
- Меняет зависимости (`npm install <pkg>`) — **по договорённости**, желательно по очереди, не оба одновременно.
- После чужого pull, если в lock конфликт — **не мёржить руками**. Проще:
  ```bash
  git checkout --theirs package-lock.json   # или origin-версию
  npm install                               # перегенерит под свою ОС
  ```
- `npm ci` для чистой установки строго по lock; `npm install` — когда осознанно меняешь зависимости.

**Итог:** native-папки (`ios/`, `android/`, `.env`) никогда не утекали и тебе ничего не ломали. Реальные конфликты были от `package-lock.json` (разные ОС) и переносов строк. Лечение: `.gitattributes` (LF) уже добавлен — сделай `git add --renormalize .`; зависимости меняем по очереди; lock при конфликте перегенерируем, а не мёржим.
